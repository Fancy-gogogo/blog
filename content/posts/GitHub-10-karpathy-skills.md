## 一、这个项目是干什么的？

### 1.1 一句话本质

**一个 Markdown 文件，装了 4 条规矩，让 AI 写代码少犯低级错误。**

整个项目的核心交付物，就是一个 `CLAUDE.md` 文件。没有框架，没有依赖，没有安装脚本。这件事本身就值得思考：为什么一个 Markdown 文件能拿到 4.4 万 Star？

### 1.2 起源：Karpathy 发了什么

项目名字里的 Andrej Karpathy，是 AI 领域公认的"最会讲课的人"之一：OpenAI 联合创始人、特斯拉 AI 总监、斯坦福 CS231n 主讲教授。他在 2026 年初的一次分享里，系统性地梳理了他用 Claude Code 的真实感受——

不是夸奖，而是**吐槽**。

他提炼出了 AI 写代码时反复出现的四个毛病：

| 毛病 | 具体表现 |
|------|---------|
| **乱做假设** | 不问你，直接猜，猜错了你还不知道 |
| **藏着困惑** | 遇到模糊需求，不澄清，自己选一个然后干 |
| **过度复杂** | 3 行能搞定的事，写出 30 行带工厂类的架构 |
| **破坏性编辑** | 改一个 bug，顺手删了注释、改了风格、动了无关代码 |

然后他说了一句话，后来被反复引用：

> *"不要告诉它做什么，给它成功标准，然后看它去做。"*
> 
> — Andrej Karpathy

开发者 forrestchang 把这些观察整理成了 4 条可执行的行为准则，放进 `CLAUDE.md` 文件，发到了 GitHub。

### 1.3 CLAUDE.md 是什么机制？

Claude Code 有一个特性：启动时会自动读取项目目录里的 `CLAUDE.md` 文件，把内容注入到系统提示词里。

这意味着——**你不需要每次对话都重复说规矩，文件放那里，它就永远记得。**

```
用户启动 Claude Code
        ↓
自动扫描项目根目录
        ↓
发现 CLAUDE.md → 注入系统提示词
        ↓
本次对话全程遵守文件里的行为准则
```

这是一个非常聪明的"寄生"设计——不改 Claude Code 本身，不需要 fork，不需要权限，只是放一个文件，就能重塑 AI 的行为模式。

### 1.4 四大原则详解

#### 原则一：Think Before Coding（编码前先思考）

**要解决的问题**：AI 在你不知情的情况下做了错误假设，等代码写完你才发现。

**核心规则**：
- 遇到模糊需求，先列出假设，询问确认
- 有多种合理解释时，呈现权衡，不要默默选一个
- 遇到真正不确定的地方，停下来，命名困惑点，主动问

**举例**：你说"优化这段代码的性能"，AI 应该先问：你说的性能是响应时间、吞吐量还是感知速度？而不是直接改代码。

#### 原则二：Simplicity First（简洁优先）

**要解决的问题**：AI 喜欢堆砌抽象，为一个简单需求造出一套复杂架构。

**核心规则**：
- 用最少代码解决当前问题，不为未来的需求提前"扩展"
- 明确禁止：添加未被要求的功能、为单次使用的逻辑创建抽象类、添加"灵活性"和"可配置性"
- 自检：一位高级工程师看到这个会觉得过度复杂吗？如果是，请简化

**举例**：你要一个折扣计算函数，AI 不应该给你一个 DiscountStrategy 工厂类，就给你一个 `calculate_discount(amount, percent)` 就够了。

#### 原则三：Surgical Changes（精准修改）

**要解决的问题**：AI 修一个 bug，顺带把整个文件"改进"了一遍，你审查代码时完全不知道哪些是为了修 bug，哪些是 AI "顺手为之"。

**核心规则**：
- 只修改必须修改的行，不"改进"相邻代码
- 匹配现有代码风格，哪怕你认为现有风格不好
- 发现无关的技术债——只提及，不删除（除非被要求）
- 如果你的修改产生了孤儿代码，清理它；但预先存在的死代码，不动

**检验标准**：差异里的每一行，都应该能直接追溯到用户的原始请求。

#### 原则四：Goal-Driven Execution（目标驱动执行）

**要解决的问题**：任务描述模糊、完成与否难以判断、AI 自认为做完了但其实没做好。

**核心规则**：
- 把模糊的任务转化为可验证的成功标准
- 多步骤任务要明确每一步的验证方法
- 优先用测试驱动：先写能复现问题的测试，再修复，再验证

**转化示例**：

| 模糊任务 | 可验证目标 |
|---------|-----------|
| "添加验证" | "为无效输入编写测试，然后让测试通过" |
| "修复 bug" | "写出能复现问题的测试，然后让它通过，确认无回归" |
| "重构 X" | "确保重构前后测试全部通过" |

## 二、为什么 Star 增长这么快？

### 2.1 增长曲线分析

| 时间节点 | Stars |

从 1 月底到 4 月中旬，这个项目已经积累了 4 万多 Star，整体处于缓慢渗透阶段。今日单日暴涨近 1 万，说明触发了某个传播临界点——最可能的原因是 Karpathy 本人或其他大 V 在社交媒体上的转发。

### 2.2 三层驱动力分析

#### 第一层：Karpathy 的名字就是流量

在 AI 开发者社区，Andrej Karpathy 的名字有独特的权威性。他不是公司代言人，不是 KOL，他是**真正做过 OpenAI、做过特斯拉 Autopilot 的人**，他说 AI 写代码有这四个问题，等于给这四个问题做了背书。

这个项目最聪明的地方是：**用他的观察，写成具体规则，放进工具里**。不是在讨论他的观点，而是直接把观点变成行动。

#### 第二层：解决了一个每天都在发生的痛苦

AI 写代码的四个毛病，不是理论问题，是**每天每个用 AI 编程的人都在受苦**的问题：

- 你以为它理解了你的需求，其实它在猜
- 你修了一个 bug，发现代码变了一堆不相关的东西
- 一个简单的工具函数，它给你搭了一套工厂模式

这些痛苦是真实的、反复的、有记忆的——任何开发者看到这四条，都会有"就是这个！"的感觉。

#### 第三层：极简主义的传播优势

这个项目的"入门成本"几乎为零：
- 不需要学习新工具
- 不需要理解复杂架构
- 复制一个文件，或者敲一行命令，立即生效

**低门槛是病毒传播的基础条件。** 当分享的成本接近于零，传播就自然发生。

### 2.3 时机因素

2026 年初到现在，正好是 Claude Code 从"极客尝鲜"进入"日常生产工具"的临界点：

- Claude Code 用户规模快速扩张
- 早期用户已经积累了足够多的"AI 写代码的挫败经验"
- 社区对"如何更好地驾驭 AI 写代码"的需求旺盛
- CLAUDE.md 机制刚好成熟

这四个条件同时满足，这个项目出现的时机恰到好处。

## 三、这个项目的价值有哪些？

### 3.1 对开发者个人：把隐性期望显性化

AI 工具的核心问题不是能力不够，而是**你的期望和它的理解之间有一条隐形的沟**。

这个项目做的事情本质上是：**把这条沟用文字填上**。

你把四条规矩写进 CLAUDE.md，AI 就有了明确的行为边界——不再猜，不再过度，不再破坏。

有开发者报告，使用这套规则后：
- Fundraising Skill 准确率从 70% 提升到 94%（+24%）
- 代码审查时间明显缩短（差异里没有"顺手改的"无关内容）
- Bug 修复过程可追溯性大幅提升

### 3.2 对团队协作：AI 行为规范的标准化载体

在团队里，一个问题是：每个人用 AI 编程的方式和质量不一样，AI 生成的代码风格和质量参差不齐。

CLAUDE.md 提供了一个解法：**把团队的 AI 编程规范写进一个文件，提交到 Git，所有人共享同一套 AI 行为准则。**

这意味着：
- AI 产出的代码质量下限有了保障
- 代码审查有了"AI 行为是否合规"的维度
- 新成员接入项目，AI 的行为自动符合团队规范

### 3.3 对行业思维：一个重要的认知升级

这个项目背后有一个很重要的认知转变：

**过去我们问"AI 能做什么"，现在我们要问"AI 应该怎么做"。**

工具能力的上限在快速提升，但工具的**行为质量**——谨慎性、精准性、可预期性——需要主动塑造。这个项目是一个信号：AI 行为工程（AI Behavior Engineering）正在成为一个需要认真对待的专业领域。

### 3.4 一个更深层的价值："提示词即代码"的范式深化

这个项目把提示词工程从"临时的、对话级的"提升到了"持久的、项目级的"：

| 传统提示词 | CLAUDE.md |
|-----------|----------|
| 每次对话手动输入 | 一次配置，永久生效 |
| 保存在聊天记录里 | 保存在 Git 仓库里 |
| 个人私有 | 团队共享 |
| 容易被长对话稀释 | 在系统层，优先级最高 |
| 不可版本管理 | 可以追踪变更历史 |

这是一个方向性的变化：**提示词正在从"用完即扔的耗材"变成"项目资产"。**

### 3.5 对 Claude Code 生态：验证插件机制的价值

这个项目同时提供了 Claude Code 插件安装方式，4.4 万 Star 的热度间接验证了：
- 开发者确实需要项目级的 AI 行为定制
- Claude Code 的插件生态有真实市场需求
- 这种"行为规则包"的模式，有可能发展成一个细分方向

## 四、从开发者视野看：这背后在发生什么？

### 4.1 AI 编程的"蜜月期"正在结束

2025 年，很多人沉浸在 Vibe Coding 的新鲜感里：自然语言描述需求，AI 瞬间生成代码，感觉无所不能。

2026 年初，这个蜜月期开始出现裂缝：

- AI 生成的代码**可以运行，但难以维护**
- 代码库在 AI 的"帮助"下变得越来越混乱
- 人类对代码的理解和掌控感在下降
- 修一个问题，引出三个新问题

Karpathy 的四条规则，本质上是在**给 AI 套上工程纪律**。这不是在限制 AI，而是在让 AI 真正融入软件工程的工作流里，而不是在工作流旁边制造混乱。

### 4.2 "规则文件"会成为 AI 工具生态的标配吗？

这个问题值得认真思考。

Claude Code 有 CLAUDE.md，Cursor 有 `.cursorrules`，GitHub Copilot 有 `copilot-instructions.md`。各家 AI 编程工具都在建立自己的"行为规则文件"机制——这不是偶然，而是一个系统性趋势。

**当工具的能力接近，行为的可预期性和可定制性就会成为竞争维度。**

### 4.3 谁能从这个趋势里获益？

这个项目触发的思考，对不同角色有不同含义：

| 角色 | 可以做什么 |
|------|----------|
| **个人开发者** | 把这套规则装进自己的项目，减少 AI 编程的无效摩擦 |
| **技术团队** | 定制团队的 AI 行为规范，把质量标准写进 CLAUDE.md |
| **工具开发者** | 围绕"行为规则文件"构建生态，提供规则市场、模板库 |
| **技术管理者** | 把 AI 行为规范纳入代码规范体系，像 eslint 规则一样对待 |

## 五、局限性与适用边界

这个项目并非没有局限：

1. **当前仅支持 Claude Code**：CLAUDE.md 机制是 Anthropic 的产品设计，对 Cursor/Copilot 需要使用各自的规则文件格式

2. **规则有效性未经严格验证**：94% 准确率等数据来自用户自报，非对照实验，实际效果因项目和使用方式而异

3. **规则可能随模型更新失效**：Claude Code 的底层行为随模型版本变化，今天有效的规则，未来不一定仍然有效

4. **需要开发者有一定判断力**：这些规则设定了行为约束，但不能替代开发者的需求描述能力。如果需求本身描述不清，规则再好也没用

5. **偏向保守**：官方说明里明确指出，这些准则"偏向谨慎而非速度"，对于需要快速原型的场景，可能会让 AI 过于谨慎
