一、场景引入：一个 HR 的真实崩溃

周一早上 9 点，互联网公司 HR 小王打开电脑，面临的是一个烂摊子：  

- 上周发布的「高级 Java 开发」岗位，在 BOSS 直聘收到 87 份简历，拉勾 52 份，猎聘 41 份，智联 28 份
    

- 总计 208 份在线简历，来自 4 个不同平台
    

- 筛选条件：3 年以上 Java 经验、本科、大厂背景、北京
    

理想情况：2 小时筛选完，推给业务部门。  
  
现实情况：  

- 4 个平台 UI 各不相同，点开每个简历都要重复操作
    

- 候选人写「精通 Java」，实际只用过 1 年 Spring MVC
    

- 有的简历是 PDF，有的是图片，有的只在网页展示
    

- 这周刚装的 ATS 系统解析脚本又崩了
    

小王的感受是：每天都在做「Ctrl C + Ctrl V」的体力活，感觉自己的工作毫无技术含量。

二、要解决这个场景，AI Agent 需要哪几项能力？

传统软件能做什么？——输入关键词，返回匹配结果。这是静态的、一次的请求。  
  
AI Agent 能做什么？——它能自主完成一个多步骤的任务，并在执行中学习和适应。  
  
从上述 HR 场景倒推，一个合格的「简历筛选 Agent」需要以下四项核心能力：

2.1 能力一：工具调用（Tools）

定义：Agent 能够调用外部工具（API、软件、硬件）完成具体任务，而不是仅仅在模型内部打转。  
  
在这个场景中：  

- 自动登录 4 个招聘平台（需要调用浏览器自动化工具）
    

- 从网页抓取简历信息（需要调用 HTML 解析工具）
    

- 把 PDF/Word/图片转成结构化文本（需要调用 OCR/文档解析工具）
    

- 将筛选结果写入 Excel 或 ATS 系统（需要调用文件/数据库工具）
    

我的核心观点：工具调用是 Agent 的「手」和「脚」，没有工具调用能力的 Agent 只是一个会说话的搜索引擎。用 Agent 的行话说：这叫「Agentic」，即 Agent 能主动操作外部世界，而不只是被动响应。  
  
技术实现：目前主流方案是 MCP（Model Context Protocol），它定义了一个统一的标准，让 Agent 能够以「插拔」的方式调用各种工具——就像 USB-C 接口一样，不同设备可以用同一个标准接入。

2.2 能力二：长期记忆（Memory）

定义：Agent 能够在多轮对话或多天运行中记住上下文信息，而不是每次都从零开始。  
  
在这个场景中：  

- 记住 HR 的偏好：「这个岗位需要 Java，不看 Python 手游履历」
    

- 记住候选人状态：「张三上周已面试，进入了二面流程」
    

- 记住历史数据：「之前标记过 3 份简历是培训机构包装，需要过滤」
    

如果没有记忆能力，Agent 就是「金鱼脑」——每次响应都是孤立的，无法形成连续的业务逻辑。  
  
我的核心观点：长期记忆是 Agent 的「脑子」。但记忆不是越多越好——很多 Agent 项目失败的原因不是「记不住」，而是「记了不该记的」。在这个场景中，HR 的反馈数据（哪些通过了、哪些拒绝了）是最重要的记忆，而不是候选人的简历原文。  
  
技术实现：目前有两种主流方案：  

1. 向量数据库（Milvus、Pinecone）：存「语义片段」，用相似度检索
    

2. 结构化存储（SQL / NoSQL）：存「业务事实」，用条件查询我的建议是混合使用——业务事实（如状态、分数）存结构化，语义知识（如偏好、标签）存向量。
    

2.3 能力三：多步推理（Reasoning）

定义：Agent 能够进行链式思考（Chain-of-Thought），把一个复杂问题拆解成多个步骤，逐步求解。  
  
在这个场景中：  

- 第一步：判断候选人是否有 Java 经验（不是「写过 Java」，而是「工作用过」）
    

- 第二步：判断经验年限是否达标（工作经历的时间线要能串起来）
    

- 第三步：判断学历是否符合（有些专科是通过成人本科进的，要能识别）
    

- 第四步：综合加权，给出匹配度评分
    

我的核心观点：多步推理是区分「聪明 Agent」和「笨 Agent」的关键。一个只会做单步匹配的 Agent（比如关键词匹配），和多步推理的 Agent 之间的差距，就像「计算器」和「数学家」的差距。  
  
一个反直觉的观察：很多 LLM 在单步推理上表现很好，但在多步推理上会「犯傻」——比如前面步骤得出的结论，到后面步骤就忘了。这是因为 Transformer 的上下文窗口有限，需要做一些工程手段来保持一致性（如分段处理、中间结果显式存储）。

2.4 能力四：规划与编排（Planning & Orchestration）

定义：Agent 能够自主制定执行计划，并根据执行结果动态调整。  
  
在这个场景中：  

- 任务分解：「先抓取 BOSS → 解析 → 匹配拉勾JD → 匹配猎聘JD → 去重 → 输出」
    

- 动态调整：「如果 BOSS 网站被封禁，切换到备用代理；如果 OCR 失败，跳过图片简历，先处理文字版」
    

- 并行执行：「4 个平台可以同时抓取，互不依赖」
    

我的核心观点：规划和编排是 Agent 的「指挥中心」。这也是目前最被低估的能力——大多数 Agent show 还停留在「单轮对话」层面，真正比拼的是「多任务编排」能力。  
  
一个务实的建议：不要一开始就想做全自动。先做「半自动」——Agent 负责抓取和解析，人类负责最终判断。跑通了再逐步增加自动化比例。

三、架构设计：用这个案例画 Agent 架构图

「用手做事」 「用脑记事」 「用脑思考」

核心模块说明：

1. 工具层（Tools）——Agent 的手足  

- 账号管理：用 Playwright 模拟登录，维持 cookie
    

- 简历抓取：每个平台一个专属解析器（Parser）
    

- OCR 处理：识别图片版简历（Qilin-OCR、PaddleOCR）
    

- 结果导出：写入 CSV、Excel，或推送到 ATS
    

2. 记忆层（Memory）——Agent 的大脑  

- 短期记忆：本轮对话的上下文（存在对话窗口）
    

- 中期记忆：本周的筛选结果（存在 Redis/MongoDB）
    

- 长期记忆：HR 的偏好、候选人标签（存在向量库）
    

3. 推理层（Reasoning）——Agent 的思考  

- CoT（Chain of Thought）：让模型把思考过程显式写出来
    

- ReAct（Reason + Act）：先推理下一步要做什么，再执行
    

- Few-shot：把 HR 之前标注的例子放在 Prompt 里作为参考
    

4. 规划层（Planner）——Agent 的指挥官  

- 任务拆分：把「筛选简历」拆成「登录→抓取→解析→匹配→去重→推送」
    

- 容错机制：每个步骤有「成功/失败/超时」三种分支
    

- 并行/串行：4 个平台可以并行抓取，但匹配要串行（受限于 API 调用次数）
    

四、难点：落地有哪些坑？

4.1 平台反爬与登录维持

问题：BOSS 直聘、拉勾们对爬虫的态度是「发现即封禁」，不会跟你商量。  
  
翻车现场：  

- 好不容易登录成功，第二天 cookie 过期了
    

- 请求频率太高，IP 被封禁
    

- UI 大版本更新，之前的 XPath 全部失效
    

我的核心观点：这不是技术问题，是商业利益的冲突。招聘平台靠「贩卖简历下载次数」赚钱，你用 Agent 高效筛选 = 断了它的财路。所以反爬不是「能不能绕过」，而是「人家愿不愿意让你过」。  
  
务实的应对：  

1. 使用 Playwright 真身模拟，别用 requests（在平台眼里你是「正常用户」）
    

2. 设置请求间隔（每份简历间隔 3-5 秒，别一口气抓 200 份）
    

3. 准备备用方案（一个账号被封，还有 3 个备用账号）
    

4.2 简历质量的识别

问题：有些候选人的简历是「纯金」，有些是「镀金」——AI 很难区分。  
  
翻车现场：  

- 候选人写「精通 Java」，实际只在培训班学过
    

- 候选人写了 3 年工作经验，但其中 1 年是「待业」，简历上写「休息」
    

- 候选人把所有接触到的技术都罗列了 20 项，其实都是培训班的练习项目
    

我的核心观点：AI 无法 100% 解决「简历注水」问题，因为「水」和「不水」本身就是主观判断。 一个 5 年的「资深工程师」写的简历，可能不如一个应届生写得漂亮。  
  
务实的应对：  

1. 设置「最低信息量阈值」：姓名+手机号+工作经历，有一个为空就标记为「待人工复核」
    

2. 让 Agent 打一个「简历可信度」分，而不是只打「匹配度」分
    

3. 把「疑似注水」的简历单独推给人做人工复核，而不是试图用 AI 完全替代人工
    

4.3 筛选准确率的瓶颈

问题：用 LLM 做匹配，看起来很美好，但准确率很难提升。  
  
翻车现场：  

- Agent 说「匹配」，HR 看了简历后发现不匹配（假阳性）
    

- Agent 说「不匹配」，但 HR 认为可以聊聊（假阴性）
    

- Agent 给出的理由很牵强：「候选人虽然经验不足，但态度积极」
    

我的核心观点：目前的 LLM 做的是「文字游戏」，而不是「招聘判断」。 它能理解文字意思，但没有人家的「行业常识」——它不知道「3 年经验的 Java 开发应该会什么」，它不知道「这条简历的包装感有多重」。  
  
几个可能有用的优化方向：  

1. 分层筛选：
    

- 第一层：规则过滤（学历、年限、薪资）——这一步解决 60% 的明显不匹配
    

- 第二层：LLM 技能匹配——这一步解决 30% 的模糊匹配
    

- 第三层：人工兜底——剩下 10% 交给人
    

1. Few-shot Learning（少样本学习）：
    

- 让 HR 手动标注 20 份「典型匹配」和「典型不匹配」的简历
    

- 把这些案例放在 Prompt 里，让 LLM 学习「这个 HR 要什么人」
    

1. 定期校准：
    

- 每周末抽出 10 份 Agent 判为「匹配」的简历
    

- 让 HR 复盘准确率，用这个数据来调模型，而不是只看 Agent 自己给出的分数
    

4.4 数据隐私与合规

问题：简历涉及候选人的个人信息，处理不当有法律风险。  
  
翻车现场：  

- 把候选人简历传到境外 LLM（如 GPT），可能涉及数据出境
    

- 未经候选人同意，把简历存到第三方数据库
    

- 在筛选中泄露候选人的年龄、性别（简历里有，但不应用作筛选依据）
    

我的核心观点：**隐私合规不是「可选项」，是「必选项」。现在行业内大家在宣传「AI 筛选」时闭口不提合规，这不是技术问题，是良心问题。  
  
务实的应对：  

1. 本地化部署：使用国产 LLM（Qwen、DeepSeek）在境内处理
    

2. 脱敏处理：在送给 LLM 之前，去除姓名、照片、身份证号等敏感字段
    

3. 合规审计：把数据处理流程写清楚，定期检查是否符合《个人信息保护法》
    

五、总结：AI Agent 能做什么、不能做什么？

回到开头 HR 小王的场景。一个理想的「简历筛选 Agent」应该做到：  

1. 自动抓取：每天早上自动登录 4 个平台，把新简历抓下来
    

2. 智能解析：把网页/PDF/图片 转成结构化数据
    

3. 语义匹配：用 LLM 判断候选人是否匹配 JD，给出理由
    

4. 及时推送：每天上午推送给 HR：「收到 X 份简历，X 份匹配」
    

AI Agent 能做的：

- 重复性劳动：登录、抓取、解析、导出，这些脏活累活 Agent 能做得比人快
    

- 规模化处理：200 份简历同时处理，Agent 不会累，人会累
    

- 一致性输出：按同一套规则筛选，不会「心情好就松一点，心情差就严一点」
    

AI Agent 不能做的：

- 招聘直觉：候选人是不是「对的人」，这个判断最终要靠人
    

- 异常处理：候选人打电话来问「什么时候面试」，这个要人接
    

- 责任承担：招错了人，这个锅 Agent 背不了
    

最后说几句得罪人的话：

很多厂商宣传「AI Agent 替代 HR」，我的观点是：替代 80% 的体力活可以，替代 20% 的判断力不可能。  
  
正确的姿势是：用 Agent 做「筛选」，让人做「决定」——这才是人机协同的正确解法。
