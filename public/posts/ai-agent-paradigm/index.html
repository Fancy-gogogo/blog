AI Agent 三种范式深度详解

三种范式：ReAct（Reasoning + Acting）、Plan-and-Execute（谋定而后动）、Reflection（反思与自我改进）

总览：为什么会有这三种范式？

Agent 的本质是让 LLM 自主完成复杂任务。但 LLM 原生是"问答机器"，不会主动行动、规划、纠错。三种范式解决的是同一个问题的三个递进层次：

层次 核心问题 范式
第一层 怎么边想边做？ ReAct
第二层 怎么先想清楚再做？ Plan-and-Execute
第三层 怎么做完之后反思变好？ Reflection

范式一：ReAct（Reasoning + Acting）

核心思想

交替推理与行动，每一步都依赖上一步的观察结果。

论文来源：2022 年 Google/Princeton《ReAct: Synergizing Reasoning and Acting in Language Models》

运作循环

Thought → Action → Observation → Thought → Action → Observation → ... → Final Answer

每一轮：

  • Thought：LLM 想"我现在应该做什么"
  • Action：调用工具（搜索、代码执行、API 等）
  • Observation：获得工具返回结果
  • 根据结果重新思考，进入下一轮

真实例子

任务：「特斯拉2024年全年营收是多少，换算成人民币是多少？」

Thought: 需要先查特斯拉2024年营收数据
Action: search("Tesla 2024 annual revenue")
Observation: Tesla 2024 revenue = $97.69 billion

Thought: 需要查当前美元兑人民币汇率
Action: search("USD to CNY exchange rate")
Observation: 1 USD = 7.25 CNY

Thought: 可以计算了
Action: calculator(97.69 * 7.25 * 1e9)
Observation: 708,252,500,000

Final Answer: 约7082亿人民币

核心价值

  • ✅ 灵活应变：每步都能根据实际结果调整方向
  • ✅ 可解释性强：推理过程透明，每步有据可查
  • ✅ 实现简单：几乎所有主流框架（LangChain、LlamaIndex）的默认模式
  • ✅ 适合短任务：步骤少、依赖实时反馈的任务性价比最高

核心缺陷

  • ❌ 长任务容易迷失：10步以上容易"偏航"，忘记最初目标
  • ❌ 步骤间无全局规划：走一步看一步，不会提前识别瓶颈
  • ❌ Token 消耗随步数线性增长：每轮都要把历史塞入上下文
  • ❌ 容易陷入循环：某些环境下会反复调用同一工具

适用场景

  • 信息检索 + 计算的简单组合任务
  • 步骤 ≤ 5 步的工具调用链
  • 需要实时环境反馈的交互式任务（如网页操作）
  • 快速原型验证

范式二：Plan-and-Execute（谋定而后动）

核心思想

先生成完整计划，再逐步执行。规划者和执行者解耦。

论文来源：2023 年《Plan-and-Solve: Improving Zero-Shot Chain-of-Thought Reasoning by Large Language Models》

运作架构

[Planner LLM] → 生成任务分解计划
      ↓
[Executor LLM/Agent] → 按计划逐步执行子任务
      ↓（可带反馈）
[Replanner] → 根据执行结果动态修正计划

关键特征：Planner 和 Executor 可以是不同的模型（Planner 用强模型，Executor 用便宜模型）

真实例子

任务：「帮我做一份竞品分析报告，对比我们的产品和市场上Top3竞品」

[Planner 输出的计划]
Step 1: 搜索市场上该品类Top3竞品名称
Step 2: 分别抓取3个竞品的核心功能、定价、用户评价
Step 3: 收集我方产品的同类数据
Step 4: 按功能/价格/口碑三个维度建立对比矩阵
Step 5: 撰写分析报告，给出结论建议

[Executor 逐步执行，每步完成后标记，发现Step2需要登录则Replanner调整]

核心价值

  • ✅ 全局视角：在执行前就能识别关键路径和潜在阻塞点
  • ✅ 长任务稳定性：不会因为中途结果而"走偏"
  • ✅ 成本可控：Planner 一次性规划，Executor 可用小模型
  • ✅ 并行潜力：计划中无依赖的步骤可以并行执行
  • ✅ 可审计：人类可以在执行前 review 计划，介入修改

核心缺陷

  • ❌ 初始计划质量决定上限：如果规划错了，执行越认真越偏
  • ❌ 对动态变化适应慢：计划制定时的假设可能在执行时失效
  • ❌ Replanner 增加复杂度：完整实现需要计划-执行-重规划三层
  • ❌ 冷启动成本高：任务开始前必须完成完整规划，有延迟

适用场景

  • 需要多步骤、有明确结构的复杂任务（报告生成、数据分析）
  • 步骤间有依赖关系需要提前协调的任务
  • 对成本敏感、希望 Planner/Executor 分离降本的场景
  • 需要人类审核计划再执行的企业级工作流
  • 任务步骤 ≥ 7 步的长链任务

范式三：Reflection（反思与自我改进）

核心思想

执行完成后（或执行中）进行自我评估，发现问题，主动改进输出质量。

论文来源：2023 年《Reflexion: Language Agents with Verbal Reinforcement Learning》

运作架构

[Actor]：执行任务，生成输出
   ↓
[Evaluator]：评估输出质量（可以是 LLM 自评，也可以是外部验证）
   ↓
[Reflector]：生成语言化的反思："哪里错了？为什么？下次怎么改？"
   ↓
[Memory]：反思结果写入 Episode Memory
   ↓
[Actor]：带着反思重新执行，循环直到满意或超出次数上限

两种 Reflection 模式

模式A：Self-Reflection（自我反思）

输出初稿 → LLM 扮演评审角色批评自己 → 根据批评改稿 → 循环

模式B：Reflexion（带记忆的反思）

第一次尝试失败 → 写下反思记录 → 下次遇到同类任务，带着上次的反思去做

真实例子

任务：「写一段 Python 代码，解析 JSON 并计算嵌套字段的平均值」

[第1轮 Actor]：生成代码
[Evaluator]：运行代码 → 报错 KeyError

[Reflector 反思]：
"代码假设所有 JSON 对象都有 'value' 字段，但实际数据中部分对象该字段缺失。
下次应加 .get() 防御性取值，并处理 None 的情况。"

[第2轮 Actor]：带着反思重写代码，加入容错处理
[Evaluator]：运行通过 ✓

[Memory 记录]：JSON 解析需注意字段缺失，用 .get() 代替直接访问

核心价值

  • ✅ 质量持续提升：通过迭代逼近"优秀"而非"勉强能用"
  • ✅ 跨任务学习：Reflexion 模式下，失败经验可以迁移
  • ✅ 减少人工干预：Agent 自己发现问题自己修复
  • ✅ 特别适合代码生成：有客观的执行反馈（报错/通过）
  • ✅ 弥补 ReAct 的短板：为 ReAct 的每轮结果加一层质量门控

核心缺陷

  • ❌ 成本倍增：每次反思都是额外的 LLM 调用，N轮反思 = N倍成本
  • ❌ 反思质量依赖模型能力：弱模型的反思可能是错的，适得其反
  • ❌ 收敛不保证：某些任务反思再多也无法解决根本问题
  • ❌ Memory 管理复杂：长期记忆如何压缩、过期是工程难题
  • ❌ 延迟高：实时性要求高的场景不适合

适用场景

  • 代码生成与调试（有客观验证标准）
  • 文案创作、报告写作（需要多轮打磨的内容）
  • 数学推理和逻辑题（有明确对错）
  • 对质量要求极高、允许多轮迭代的任务
  • 可以积累经验的长期 Agent 系统

如何选型？一张决策图

任务复杂吗？
├── 否（≤3步）→ 直接用 ReAct，简单高效
└── 是
    ├── 有明确结构/阶段划分？
    │   ├── 是 → Plan-and-Execute（先规划，再执行）
    │   └── 否 → ReAct（动态探索）
    │
    └── 对输出质量要求高？允许多轮迭代？
        ├── 是 → 加上 Reflection 层（包在 ReAct 或 P&E 外面）
        └── 否 → 保持 ReAct / P&E 即可

⚠️ 重要认知：三种范式不是互斥的，而是可以嵌套组合的 最强 Agent = Plan-and-Execute 的骨架 + ReAct 的每步执行 + Reflection 的质量保障

一句话记住它们

范式 一句话
ReAct 见招拆招，灵活但容易走偏
Plan-and-Execute 运筹帷幄，稳健但计划可能过时
Reflection 精益求精，高质量但代价是时间和成本
